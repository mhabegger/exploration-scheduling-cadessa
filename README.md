# Cadessa

> **Every track, right on cue.**
> Human taste. Machine patience.

Cadessa is an explainable music-programming studio for playlists, linear radio, and produced shows. It turns a format strategy into a timed running order, explains every choice, preserves every revision, and gives a programmer fast control over holds and replacements.

This repository now contains a working, credential-free vertical slice—not just a visual concept. Open the studio, generate a schedule on the Worker, hold songs, reflow around those holds, compare and restore preserved revisions, replace a song from ranked alternatives, approve the result, and download CSV, JSON, or M3U exports. The state is persisted per Channel in a Cloudflare Durable Object.

Live Spotify/Deezer publishing, playout delivery, Postgres persistence, and audio storage are deliberately inactive until their credentials, resource IDs, and music rights are supplied.

[Open the short standalone introduction](./introduction.html), or run the studio locally.

## What works now

- **Flow Studio** — loads a server-persisted schedule, reflows it, holds/releases tracks, offers ranked replacements, approves revisions, restores history, and downloads exports.
- **Conductor 0.2** — a strict TypeScript, deterministic scheduler with rotation clocks, time-based artist/title separation, duration targeting, history-aware scoring, pinned tracks, and decision evidence.
- **Real Hot AC metadata** — 60 actual current, recurrent, discovery, and gold titles based on the [American Top 40 Hot AC chart](https://www.americantop40.com/charts/hot-ac-243/latest/) and [Mediabase year-end context](https://pdf.mediabase.com/yearend25/YearEnd2025_US.pdf).
- **Library** — searchable/filterable catalog inspection with honest source capability and metadata provenance.
- **Insights** — metrics calculated from the active persisted revision rather than invented audience data.
- **Durable jobs** — alarm-driven, chunked generation carries play history across hour and segment boundaries.
- **Cloud-ready boundaries** — Drizzle/PlanetScale, Hyperdrive, R2, provider adapters, and OpenRouter are isolated behind explicit capabilities.
- **Introduction** — an executive overview at `/introduction` and a standalone `introduction.html` for easy sharing.

The song titles and artist names are factual scheduling metadata. Durations, BPM, energy, and moods are labeled editorial planning estimates. Abstract gradients are used instead of provider artwork. No audio, provider IDs, playlist access, broadcast rights, or master ownership is claimed.

## The working workflow

```text
REAL CATALOG → CONDUCTOR → VALIDATE → SAVE REVISION → HOLD / REPLACE → APPROVE → EXPORT
                       Durable Object is the local canonical schedule store
```

1. The first request for a Channel creates a deterministic schedule and revision 1.
2. Conductor fits a 12-position Hot AC rotation pattern to a 44-minute music window.
3. Eligibility enforces destination capability, configured exclusions, known-explicit flags, title rest, and every credited artist’s rest in elapsed minutes. Unverified master ratings remain visibly unknown.
4. Scoring balances energy progression, tempo transitions, rotation need, era variety, discovery share, and target duration.
5. Reflow creates revision N+1 while preserving held positions.
6. Alternatives are generated by re-running the same constraints with the rest of the schedule pinned; a replacement creates another preserved revision.
7. Approval changes the active revision state. Restore never destroys history—it creates a new revision derived from the selected version.
8. CSV, JSON, and metadata-only M3U downloads are generated on the server. Missing masters are explicit `missing://` entries, not fake playable URLs.

## Technical architecture

```text
Browser
  └─ React 19 + TanStack Start + Tailwind + shadcn-compatible UI
      └─ Cloudflare Worker (SSR + server routes)
          ├─ Conductor deterministic scheduling domain
          ├─ ScheduleCoordinator Durable Object
          │   ├─ preserved studio revisions
          │   └─ alarm-driven multi-hour jobs
          ├─ Drizzle → pg → Hyperdrive → PlanetScale Postgres
          ├─ private R2 boundary for owned/licensed audio
          ├─ Spotify / Deezer / broadcast adapters
          └─ OpenRouter boundary for permitted interpretation/explanation
```

The working studio uses a Durable Object as its low-latency schedule coordinator. The Worker now also has an invocation-scoped Drizzle/`pg` connection boundary for PlanetScale Postgres through Hyperdrive, plus a database-readiness endpoint. Moving the remaining studio documents into the included Postgres schema can happen incrementally without changing the UI contract.

### Core choices

- React 19 and TanStack Start with file-based routes and full-document SSR.
- Tailwind CSS v4 plus shadcn-compatible source components in `src/components/ui`.
- TypeScript strict mode; type checking is a separate required build step.
- One `ScheduleCoordinator` Durable Object per Channel.
- Alarm work is idempotent; checkpoint and segment keys make retries safe.
- PlanetScale Postgres via Cloudflare Hyperdrive and `pg`, modeled with Drizzle.
- Private R2 keys for customer-owned or licensed media only.
- OpenRouter receives only user-authored, Cadessa-generated, or permitted owned metadata.

## Run locally

Requirements: Node.js 22.12+ and pnpm 11+.

```bash
pnpm install
pnpm dev
```

Open `http://localhost:3000`. The schedule workflow and exports still work if Postgres is temporarily unavailable. To exercise the database locally, copy `.env.example` to `.env`, set a local connection string, and keep the placeholder Hyperdrive ID for local emulation only.

Useful checks:

```bash
pnpm typecheck
pnpm test
pnpm build
pnpm check
```

Database commands:

```bash
pnpm db:generate
pnpm db:migrate
pnpm db:studio
```

`DATABASE_URL` must be a direct PlanetScale Postgres branch URL for Drizzle Kit. Migrations do not run through a Worker binding and never run during Worker startup.

## API surface

| Endpoint | Method | Purpose |
| --- | --- | --- |
| `/api/health` | GET | Worker health, environment, and engine version |
| `/api/health/database` | GET | Probe Postgres through Hyperdrive and verify that the initial schema is migrated |
| `/api/schedule` | GET / POST | Generate a synchronous deterministic schedule |
| `/api/studio/:channelId` | GET | Read the active revision and history; add `?format=csv`, `json`, or `m3u` to download |
| `/api/studio/:channelId` | POST | Generate, hold, list alternatives, replace, approve, or restore |
| `/api/studio/:channelId` | DELETE | Reset that Channel’s local studio revisions |
| `/api/generation/:channelId` | POST | Enqueue or resume an alarm-driven horizon generation |
| `/api/generation/:channelId` | GET | Read its checkpoint, real progress, and ready-result summary; add `?include=schedules` for generated hourly music windows |
| `/api/generation/:channelId` | DELETE | Cancel the active generation |

Example actions:

```json
{ "action": "generate", "seed": "afternoon-refresh" }
{ "action": "hold", "position": 0, "trackId": "hotac-08-golden" }
{ "action": "alternatives", "position": 5 }
{ "action": "replace", "position": 5, "trackId": "<an id returned by alternatives>" }
{ "action": "approve" }
{ "action": "restore", "revision": 1 }
```

Hold requests are optimistic-concurrency checks: `trackId` must match the item at that position in the active revision. Replacement IDs should be selected from the immediately returned alternatives list.

Production authentication, workspace authorization, CSRF protection, and rate limiting should be added once an identity provider is selected.

## Provision cloud resources

Infrastructure-as-code lives in `wrangler.jsonc`. The Durable Object is declared there directly. Because a Hyperdrive configuration ID belongs to one Cloudflare account, the Cloudflare Vite plugin injects the `HYPERDRIVE` binding from the `HYPERDRIVE_ID` build variable into the generated Wrangler deployment configuration.

### Deploy the credential-free demo with Workers Builds

Connect the repository to a Worker named `exploration-scheduling-cadessa` and use:

| Setting | Value |
| --- | --- |
| Production branch | `main` |
| Build command | `pnpm run build` |
| Deploy command | `pnpm exec wrangler deploy` |
| Root directory | `/` |
| Build variable | `PNPM_VERSION=11.6.0` |
| Build variable | `HYPERDRIVE_ID=<32-character Cloudflare configuration ID>` |

No database credential is stored in the Worker or repository: Hyperdrive owns the PlanetScale origin credentials and gives the Worker a generated `connectionString`. The Durable Object namespace and its first SQLite migration are created by Wrangler during deployment. Keep custom routes empty; `workers_dev` and preview URLs are enabled in `wrangler.jsonc`. Your account must have its one-time `workers.dev` subdomain configured in the Cloudflare dashboard.

The Workers Builds API token must include account-level **Hyperdrive Read** in addition to its normal Workers deployment permissions. Use **Hyperdrive Write** only for the one-time create/update command; the application build does not create database infrastructure.

The Worker name in Cloudflare must match the `name` in `wrangler.jsonc`. If you use a differently named Worker, change that one field before connecting the build.

1. Create a PlanetScale Postgres database, an application role, and a separate migration role. PlanetScale requires TLS; use the provided URL with `sslmode=verify-full`.
2. Create Hyperdrive once, with query caching disabled so administrative reads retain read-after-write behavior:

   ```powershell
   $env:DATABASE_URL = "postgresql://.../cadessa?sslmode=verify-full"
   pnpm exec wrangler hyperdrive create cadessa-postgres --connection-string="$env:DATABASE_URL" --caching-disabled
   ```

3. Copy the returned 32-character configuration ID into the Cloudflare Workers Builds variable `HYPERDRIVE_ID`. It is a resource identifier, not the database password.
4. Apply Drizzle migrations separately with the direct migration-role URL. Drizzle Kit cannot use a Worker binding, and migrations are intentionally not run during Worker startup or every preview deployment:

   ```powershell
   $env:DATABASE_URL = "postgresql://.../cadessa?sslmode=verify-full"
   pnpm db:migrate
   ```

5. Deploy, then verify `GET /api/health/database`. A `200` response with `status: "ready"` proves the Worker reached the migrated database through Hyperdrive. `migration_required`, `unavailable`, and `unconfigured` responses return `503` without leaking connection details.
6. Create private R2 buckets named `cadessa-audio` and `cadessa-audio-preview`, then add the `AUDIO_BUCKET` binding when licensed masters are available.
7. Add provider OAuth applications and encrypted-token storage after the Worker hostname and identity model exist.
8. Verify and deploy:

   ```bash
   pnpm check
   pnpm deploy
   ```

Cloudflare bindings do not inherit into named Wrangler environments; repeat Hyperdrive, R2, Durable Object, and variable configuration for each environment.

## Data model

The generated SQL migration in `drizzle/` models workspace membership, Channels, provider capabilities, canonical artists and recording groups, tracks and assets, versioned Sound DNA, clocks and slots, generation checkpoints, versioned schedules, resumable publications, audit history, and provenance-tagged LLM artifacts.

All instants use `timestamptz`, Channels retain an IANA timezone, and durations use integer milliseconds. Runs freeze their engine version, seed, input hash, catalog watermark, profile version, clock version, and horizon for reproducibility.

## Scheduling guarantees in this build

- deterministic results for the same validated input and seed;
- destination capability and known-explicit metadata filtering, with unknown master ratings preserved;
- track uniqueness plus elapsed-time artist and title separation;
- credited/featured artists share the same rest checks;
- prior-play history carries across alarm chunks;
- a target-duration score and measured duration-fit health;
- rotation-pattern fit, energy arc, era balance, discovery, and tempo scoring;
- holds survive server-side reflow; the pure engine also accepts explicit exclusions;
- alternatives are revalidated against the complete pinned schedule;
- compact evidence is stored with each selected item;
- impossible inputs return a safe partial schedule instead of silently violating a hard rule.

Fixed speech/commercial anchors, local music-research ingestion, statutory quota packs, DST clock fixtures, licensed-file analysis, playout reconciliation, authentication, and real provider delivery remain credential- or product-specific extensions.

## Provider and AI boundaries

Spotify is a capability-limited personal-playlist destination, never a source of commercial broadcast audio. Cadessa does not download Spotify or Deezer audio to R2, fabricate provider IDs, or send restricted provider content to OpenRouter. Broadcast delivery uses only customer-owned/licensed R2 assets or explicit playout references.

OpenRouter is limited to translating user-authored briefs into validated proposals, explaining deterministic score evidence, and optionally tagging customer-owned metadata. Its output is schema-validated and cannot override invariants, licensing, or final ordering.
